踩坑：阿里云 authorized_keys 写入格式错误

问题：网页终端复制粘贴时，authorized_keys 被拆成 authorized keys（带空格），公钥被断行。

解决：用 heredoc 逐行写入（不要复制粘贴长行）：

踩坑：ssh -o 参数拼写错误

问题：-o StrickHostKeyCheck=no → Bad configuration option

解决：正确的参数是 StrictHostKeyChecking。

踩坑：网页终端 heredoc 自动缩进

问题：网页终端在 cat > file << 'EOF' 后自动给每行加缩进，导致 Python/Shell 脚本语法错误。

解决：用 printf '%s\n' 逐行写入文件，或接受 yes 后直接 SSH 免密。

推理速度

比 0.5B 慢 3-4 倍，单次 RAG 回答等待超过 30 秒——评委无法接受。

知识精度

图文混排

两个本地模型均无法正确输出后端要求的 blocks 图文交替格式，图文混排功能完全失效。

结论

1.5B 在龙芯 3A5000 上既慢又笨，不适合生产使用。 不推荐在无 SIMD 加速的龙芯 CPU 上运行 1.5B 级别的通用 LLM 做专业领域 RAG。除非有领域微调模型或 0.5B 级的专门检索模型，否则本地推理仅具验证意义。

当前策略：DeepSeek API 为主线（保证演示质量），llama.cpp 编译成功 + 本地推理调通作为国产算力亮点展示保留。

九、关键经验教训

  1. 龙芯 3A5000 纯 CPU 推理结论——太小太蠢，太大太慢：
    • 0.5B：7-8 t/s，速度快但知识为 0——幻觉严重（胡编”文心一白”），完全不可用
    • 1.5B：极慢（>30s 单次），答案错误（手册 18-22 N·m 被答成 10-14 N·m），图文混排失效
    • 7B：理论上可装（~4.5GB），但在 4 核 2GHz 无 SIMD 加速条件下预计 0.3-0.5 t/s，评委等不起
    • 结论：无向量加速的通用 CPU 不适合跑通用 LLM 做专业领域 RAG；llama.cpp 编译成功保留为国产算力验证，生产走 DeepSeek API
  2. 网页终端 heredoc 有坑：自动缩进导致 Python/Shell 脚本语法错误（IndentationError），用 printf '%s\n' 逐行写文件替代
  3. systemd ExecStart 不能换行：长命令必须写在一行，否则启动失败 status=1/FAILURE
  4. SSH 隧道 GatewayPorts 限制：ssh -L 0.0.0.0:port 需要 sshd 配置 GatewayPorts yes；绕过方法是反向隧道到跳板 + 本地 ssh -L 拉回
  5. 反向 SSH 隧道是内网穿透的最简方案：只要目标能出站，不需要任何额外软件
  6. 大文件用跳板中转：龙芯 VM 直下 1.5B 仅 87 kB/s → 5090 跳板下载数 MB/s → SCP 到龙芯 615 KB/s，虽然也不快但比直下快 7 倍
  7. mirrors 策略：GitHub 不通时，hf-mirror.com 下载模型，gitee 拉代码，Windows 打包中转上传也是可行方案
  8. 不要在网页终端粘贴多行命令：命令全粘在一起不执行；逐条执行，每步确认输出
  9. Turbopack rmdir EBUSY：Windows 上 out/ 被 Explorer/SSH 进程锁住时，Next.js 构建报 EBUSY: resource busy or locked。解决：用 cmd /c 跑构建、或先 rmdir /s /q out 再构建
  10. Nginx 超时 + 浏览器缓存组合拳：本地 LLM >30s 响应 → Nginx 默认 60s 超时不够 → 前端收到空响应显示 404 → 浏览器缓存旧 JS → 新代码不生效。解决：proxy_read_timeout 120s + Cache-Control: no-cache + Ctrl+Shift+R

十、前端模型切换功能

10.1 需求

在 AI 助手面板添加一个 [云端] [本地] 切换按钮，让评委既能体验 DeepSeek 的精准回答，又能看到龙芯国产 CPU 本地推理的验证成果。

10.2 前端改动 (ai-assistant.tsx)

  • 新增 ModelMode 类型和 modelMode state（默认 "cloud"）
  • CardHeader 右侧加 segmented control：两个按钮 Cloud/Cpu 图标 + “云端”/“本地” 文字
  • 选中态 bg-primary text-primary-foreground，非选中态透明
  • POST /api/chat 时 body.model = modelMode
  • 状态指示文字跟随切换：● 云端模型 / ● 本地模型

10.3 后端改动 (main_stdlib.py)

  • 新增 LOCAL_LLM_BASE (默认 http://127.0.0.1:8080) 和 LOCAL_LLM_MODEL 配置
  • 新增 _handle_local_chat() 方法：复用检索 prompt，调 llama.cpp OpenAI 兼容 API
  • do_POST 读 body.model："local" → _handle_local_chat，默认 → DeepSeek
  • 本地模式：跳过图片混排（模型太弱，输出不了章节引用），timeout 120s

10.4 部署踩坑

构建：Windows 上 Turbopack 报 EBUSY 锁 out/ 目录，PowerShell Remove-Item -Force 也删不掉。最终用 cmd /c "rmdir /s /q out & set NEXT_PUBLIC_API_BASE= & npx next build" 绕过。

404 排查：切本地模式后浏览器报 404，但 curl 直接打 Nginx 80 端口正常返回。排查链路：

  1. systemctl status llama-server → 正常 running
  2. curl 8080/health → {"status":"ok"}
  3. curl 8001/api/chat -d 'model:local' → 正常返回
  4. curl 80/api/chat -d 'model:local' → 正常返回
  5. 浏览器 → 404

根因：Nginx 默认 proxy_read_timeout 60s，1.5B 推理超过 60s 后 Nginx 断开连接返回空响应。前端缓存旧 JS → 没加载新代码 → 请求没带 model 字段。双重问题。
